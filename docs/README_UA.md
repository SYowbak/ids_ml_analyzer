## 1. Загальний опис системи

**IDS ML Analyzer** (Intrusion Detection System Machine Learning Analyzer) — це прикладна програмна система для автоматизованого виявлення аномалій у мережевому трафіку та журналах безпеки з використанням алгоритмів машинного навчання.

Система реалізована у вигляді локального веб-застосунку з графічним інтерфейсом, що дозволяє аналітику безпеки виконувати повний цикл роботи: від завантаження та підготовки даних — до навчання моделі, сканування вхідного файлу та інтерпретації результатів.

Ключова властивість системи — **строгий контроль доменної сумісності**: кожна модель прив'язана до конкретного формату та сімейства даних (CIC-IDS, NSL-KDD або UNSW-NB15). Система відхиляє несумісні комбінації файл/модель ще до початку обчислень, що унеможливлює некоректні результати детекції.

## 2. Мета та задачі

### 2.1 Мета

Розробити програмну систему, яка дозволяє аналітику кіберзахисту швидко та відтворювано виявляти аномалії в мережевому трафіку та журналах безпеки на основі попередньо навчених ML-моделей, без необхідності написання коду.

### 2.2 Функціональні задачі

- Завантаження та валідація датасетів у форматах CSV та PCAP/PCAPNG
- Автоматичне визначення сімейства даних (CIC-IDS, NSL-KDD, UNSW-NB15)
- Навчання моделей ML з підбором гіперпараметрів та калібруванням порогів
- Класифікація записів як «норма» / «аномалія» та мультикласова детекція типу атаки
- Формування звітів із ризик-оцінкою, топ-джерелами активності та планом дій
- Збереження історії сканувань у локальній базі даних
- Керування збереженими моделями (перегляд, порівняння метрик, видалення, завантаження)

### 2.3 Нефункціональні вимоги

| Вимога | Значення |
|--------|---------|
| Операційна система | Windows 10+, Linux, macOS |
| Мова інтерфейсу | Українська |
| Обробка великих файлів | Потоковий режим, до 2 ГБ на сесію |

## 5. Опис модулів

### 5.1 Core Layer

#### `data_loader.py` — Завантаження даних

Реалізує завантаження файлів у форматі CSV та PCAP/PCAPNG.

**Ключові можливості:**
- Чанкова обробка великих CSV без завантаження в пам'ять цілком
- Витяг мережевих ознак із PCAP-пакетів: обчислення потоків (flows) за 5-tuple `(src_ip, dst_ip, src_port, dst_port, protocol)`
- Очищення заголовків (видалення пробілів, нормалізація регістру)
- Обмеження `max_rows` для попереднього перегляду та діагностики
- Алгоритм Велфорда для онлайн-обчислення статистик без накопичення даних

#### `preprocessor.py` — Препроцесинг

Підготовка ознак перед подачею до моделі.

**Операції:**
- Видалення ознак-утечок (колонки, що безпосередньо ідентифікують атаку)
- Категоріальне кодування (`LabelEncoder`)
- Масштабування числових ознак (`RobustScaler` для Isolation Forest)
- Заповнення пропущених значень медіаною

#### `model_engine.py` — Рушій моделей

Центральний модуль навчання та інференсу.

**Підтримувані алгоритми:**

| Алгоритм | Клас задачі | Режим навчання |
|----------|-------------|----------------|
| Random Forest | Класифікація | Supervised |
| XGBoost | Класифікація | Supervised |
| Isolation Forest | Детекція аномалій | Unsupervised |
| Two-Stage (RF+RF або XGB+XGB) | Гібридна класифікація | Supervised |

> **Примітка:** `TrainingConfig` валідує 3 базових алгоритми (Random Forest, XGBoost, Isolation Forest). Two-Stage є окремою архітектурою, що активується через `TwoStageModel` поза межами `TrainingConfig`.

**Структура збереженої моделі (`.joblib`):**
```python
{
    "model":     <навчений об'єкт моделі>,
    "metadata":  {
        "algorithm": str,
        "dataset_type": str,           # "CIC-IDS" | "NSL-KDD" | "UNSW-NB15"
        "trained_families": list,
        "compatible_file_types": list,
        "two_stage_mode": bool,
        "schema_mode": str,            # "unified" | "domain"
        "two_stage_threshold_default": float,
    },
    "feature_names": list[str],
    "label_encoder": <LabelEncoder або None>,
}
```

#### `two_stage_model.py` — Двоетапна модель

Реалізує гібридну архітектуру детекції:
1. **Перший етап (Gate):** бінарний класифікатор RF/XGB — визначає «норма» чи «аномалія»
2. **Другий етап (Refinement):** мультикласовий класифікатор — визначає точний тип атаки лише для аномалій

Знижує кількість хибних тривог (FP) порівняно з чисто мультикласовим підходом.

#### `threshold_policy.py` — Стратегії порогів

Визначає оптимальний поріг активації аномалії після навчання. Профілі:
- **Збалансований (balanced):** баланс між чутливістю (recall) та специфічністю
- **Строгий (strict):** знижена чутливість, менше хибних тривог

#### `feature_adapter.py` — Адаптер ознак

Вирівнює вектор ознак файлу сканування до контракту ознак збереженої моделі. Додає відсутні колонки із нульовим значенням, відкидає зайві — зберігаючи строгий порядок ознак.

#### `dataset_detector.py` — Визначення сімейства даних

Автоматично визначає сімейство датасету за іменами колонок через голосування між доменними профілями. Повертає рівень впевненості (`confidence`) та ознаку неоднозначності (`ambiguous`).

#### `threat_catalog.py` — Каталог загроз

База знань типів атак із описами, оцінкою критичності та планами реагування. Використовується `scan_renderer.py` для генерації детальних карток загроз у звіті.

### 5.2 Service Layer

#### `scanning_service.py` — Сервіс сканування

Оркеструє повний пайплайн аналізу вхідного файлу:

```
Вхідний файл (CSV/PCAP)
        |
        v
  DataLoader.load_file()
        |
        v
  Preprocessor.transform()
        |
        v
  FeatureAdapter.align()
        |
        v
  Model.predict() / predict_proba()
        |
        v
  ThresholdPolicy.apply()
        |
        v
  Агрегація: risk_score, anomalies_count, top_sources
        |
        v
  DatabaseService.save_scan()
```

#### `training_service.py` — Сервіс навчання

Оркеструє навчання нової моделі:
- Завантаження та препроцесинг тренувального датасету
- Пошук оптимального алгоритму (AutoML-режим) або використання вибраного
- Навчання, калібрування порогів, збереження артефактів
- Генерація метрик якості (Точність, Прецизійність, Повнота, F1-оцінка)

#### `database.py` — Сервіс бази даних

CRUD-операції через SQLAlchemy ORM:
- `save_scan()` — збереження завершеної сесії сканування
- `get_history()` — отримання останніх N сканувань у вигляді словників
- `get_scans_count()` — загальна кількість виконаних сканувань
- `clear_scan_history()` — очищення всієї history (каскадно: alerts → detected_anomalies → sessions)
- `save_analysis_session()` — створення нової сесії зі статусом 'processing'
- `update_analysis_session()` — оновлення стану сесії

#### `visualizer.py` — Сервіс візуалізації

Генерація Plotly-графіків для:
- Часової хронології виявлених аномалій
- TOP джерел мережевої активності (IP/порти)
- Розподілу ризиків за сканування

---

### 5.3 UI Layer

#### `app.py` — Точка входу

Ініціалізує сервіси, налаштовує логування через Loguru, визначає 5 вкладок Streamlit та делегує рендеринг відповідним модулям.

#### `scan_renderer.py` — Рендерер результатів

Генерує HTML-звіт сканування. Усі дані користувача перед вставкою в HTML проходять через `escape_html()` — єдину точку входу захисту від XSS.

**Секції звіту:**
1. Зведення: ризик %, кількість аномалій, задіяна модель
2. Топ-загрози: картки з описом, рівнем критичності та планом реагування
3. Мережеві індикатори: топ IP-джерел, порти, протоколи (для NIDS)
4. Поведінкові індикатори: протокол, сервіс, стан сесії (для SIEM)
5. Хронологія: часовий розподіл аномалій
6. Таблиця аномалій: детальний список з обмеженням для UI

#### `scan_diagnostics.py` — Попередня діагностика

До початку сканування перевіряє:
- Сумісність розширення файлу та моделі
- Збіг доменних ознак (колонки файлу vs контракт моделі)
- Якість датасету: частка пропущених значень, дублікати, дисбаланс класів
- Очікувану частку аномалій на preview-вибірці
- Попередження якщо модель не виявляє аномалій на файлі з відомими мітками атак

#### `session_cache.py` — Кеш між вкладками

Parquet-based кеш для передачі великих DataFrame між вкладками Streamlit без зберігання в `session_state` (що спричиняє `MemoryError` при датасетах понад 300 МБ). Використовує атомарний запис (write-then-rename), TTL-очищення та sweep осиротілих сесій. Ліміт сесії: 2 ГБ.

## 6. Схема бази даних

База даних SQLite (`ids_history.db`) містить 5 таблиць (2 основних + 3 допоміжних):

### Таблиця `analysis_sessions` — Сесії аналізу

| Колонка | Тип | Опис |
|---------|-----|------|
| `id` | INTEGER PK | Ідентифікатор сесії |
| `filename` | TEXT | Назва проаналізованого файлу |
| `file_type` | TEXT | Розширення файлу (.csv, .pcap, .pcapng) |
| `upload_path` | TEXT | Шлях до завантаженого файлу |
| `file_size` | INTEGER | Розмір файлу в байтах |

## 7. Алгоритми машинного навчання

### 7.1 Random Forest (Випадковий ліс)

Ансамблевий метод, що будує набір вирішальних дерев на bootstrap-вибірках. Кінцева відповідь визначається більшістю голосів дерев. Є основним алгоритмом для задач класифікації мережевих вторгнень у системі.

**Переваги для IDS:**
- Висока стійкість до викидів і шумових ознак
- Нативна підтримка мультикласової класифікації
- Не вимагає нормалізації числових ознак

### 7.2 XGBoost (Градієнтне підсилення)

Реалізація градієнтного підсилення з регуляризацією. Будує дерева послідовно — кожне наступне мінімізує залишкову помилку попереднього. Зберігається у форматі `.ubj` для сумісності між версіями.

**Переваги для IDS:**
- Краща точність на незбалансованих класах
- Вбудований механізм регуляризації (L1/L2) проти перенавчання

### 7.3 Isolation Forest (Ліс ізоляції)

Алгоритм аномалійної детекції без учителя. Будує дерева ізоляції: аномальні точки ізолюються за меншу кількість розбиттів. **Єдиний алгоритм у системі, який підтримує аналіз PCAP-файлів** — не потребує міток класів.

Параметр `contamination` задає очікувану частку аномалій. Система підтримує режим `"auto"`.

### 7.4 Двоетапна модель (Two-Stage)

Складається з двох послідовних класифікаторів:
- **Етап 1 (Gate):** бінарний RF/XGB — «норма» або «аномалія»
- **Етап 2 (Refinement):** мультикласовий — точний тип атаки лише для аномалій

Архітектура знижує кількість FP порівняно зі звичайним мультикласовим підходом.

### 7.5 Метрики якості

| Метрика | Формула | Значення для IDS |
|---------|---------|-----------------|
| **Точність (Accuracy)** | (TP+TN) / (TP+TN+FP+FN) | Загальна частка правильних рішень |
| **Прецизійність (Precision)** | TP / (TP+FP) | Частка реальних атак серед знайдених |
| **Повнота (Recall)** | TP / (TP+FN) | Частка знайдених атак з усіх реальних |
| **F1-оцінка** | 2·P·R / (P+R) | Баланс між Precision та Recall |

> **Важливо:** для IDS-задач Recall критичніший за Precision — пропущена атака (FN) небезпечніша за хибну тривогу (FP).

## 11. Безпека та захист

### 11.1 Захист від XSS

Усі рядки з користувацьких даних перед вставкою в HTML-шаблони проходять через `escape_html()`:

```python
import html

def escape_html(value: Any) -> str:
    """Єдина точка входу для всіх ненадійних даних у HTML-блоках."""
    return html.escape(str(value), quote=True)
```

Екрануються символи: `&`, `<`, `>`, `"`, `'`.

### 11.2 Безпека файлових операцій

- Завантажені файли зберігаються з UUID-префіксом (`scan_<uuid>_<назва>`) для уникнення зіткнень та path traversal атак
- Видалення моделей виконується через `_safe_model_path()` з валідацією шляху відносно директорії `models/`

### 11.3 Конфігурація безпеки Streamlit

```toml
# .streamlit/config.toml
[server]
enableCORS = false
enableXsrfProtection = false
```

> `enableXsrfProtection = false` є стандартним налаштуванням для локального розгортання Streamlit. При публічному розгортанні за reverse proxy необхідно включити захист та налаштувати `allowedOrigins`.

| Термін | Визначення |
|--------|-----------|
