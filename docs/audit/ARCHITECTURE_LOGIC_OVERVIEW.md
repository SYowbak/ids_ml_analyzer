# Огляд архітектури і логіки

## Активний runtime-поверх
Фактичний активний застосунок запускається через `src/ui/app.py` і використовує 5 вкладок:
- Головна
- Тренування
- Сканування
- Моделі
- Історія

## Реальний потік виконання
1. Користувач обирає файли та параметри у вкладці тренування.
2. Training-пайплайн формує модель і зберігає bundle + metadata у `models/`.
3. Сканування підтягує модель, перевіряє сумісність файлу/схеми, виконує інференс.
4. Результат збагачується risk/severity/recommendation логікою.
5. Summary сканування зберігається в SQLite через `DatabaseService`.

## Домени даних
- `CIC-IDS` — NIDS-потік, CSV/PCAP (залежно від моделі).
- `NSL-KDD` — SIEM-подібний CSV-домен.
- `UNSW-NB15` — сучасний SIEM/мережевий CSV-домен.

## Ключові контролі безпеки
- Контракт ознак і schema-перевірка перед transform.
- Перевірка сумісності моделі та файлу перед запуском.
- Контроль порогу детекції (threshold/sensitivity).
- Калібрування IF-порогу з metadata.
- Обмеження семплу/балансування для стабільності навчання.

## Фактичні структурні ризики (стан після аудиту)
- Паралельні активні/неактивні service-path гілки в репозиторії.
- Наявність евристик у scan-threshold політиках, що потребують уніфікації provenance.
- Критичні E2E тести історично були skippable без strict-профілю.

## Що вже виправлено в коді
- Safe auto-fallback у виборі моделі для scan.
- Strict CSV precheck (missing + unexpected).
- Підтвердження збереження scan в історію (`history_saved`).
- `ModelEngine.auto_calibrate_isolation_threshold(...)` для service-path сумісності.
- Нормалізація DB-path і rebind engine при зміні `db_path`.
