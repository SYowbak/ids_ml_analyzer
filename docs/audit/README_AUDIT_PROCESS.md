# IDS ML Analyzer — Процес повного аудиту

## Мета
Провести відтворюваний аудит проекту по всіх критичних площинах:
- архітектура і активні runtime-шляхи;
- коректність навчання та інференсу ML-моделей;
- IDS-логіка і ризики хибнонегативів/хибнопозитивів;
- надійність збереження результатів;
- якість тестового контуру;
- відповідність документації фактичній поведінці застосунку.

## Як проводився аудит
Аудит виконувався ітеративно у спринтах (Sprint 01-08) з фіксацією артефактів у `docs/audit/`.
Кожний спринт містив:
1. Перевірку конкретного блоку коду.
2. Формалізовані висновки з пріоритетом ризику.
3. Верифікацію тестами або runtime-перевірками.
4. Оновлення трекера покриття та наступних кроків.

## Ключові артефакти
- `AUDIT_TRACKER.md` — стан покриття, прогрес remediation.
- `FILE_INVENTORY.md` — інвентар файлів в рамках аудиту.
- `ARCHITECTURE_LOGIC_OVERVIEW.md` — логіка активного runtime-потоку.
- `DIAGRAMS.md` — Mermaid-діаграми реальних потоків.
- `FINDINGS_SPRINT_XX_*.md` — детальні висновки по спринтах.
- `FINAL_AUDIT_REPORT.md` — консолідований фінальний звіт.
- `REMEDIATION_ROADMAP.md` — пріоритизований план виправлень.

## Модель доказів
Кожний висновок валідний лише якщо містить:
- конкретне місце в коді;
- пояснення поведінки;
- операційний вплив;
- напрям виправлення.

## Шкала критичності
- `CRITICAL` — ризик зриву довіри до детекції, серйозний пропуск атак або runtime-break.
- `HIGH` — суттєва деградація якості/надійності.
- `MEDIUM` — технічний борг із високою ймовірністю регресій.
- `LOW` — локальні проблеми підтримуваності або UX-читабельності.

## Поточний стан
Аудит завершено; проект перейшов у фазу практичного remediation з уже внесеними кодовими фіксами.
Поточний status та зроблені кроки див. у `AUDIT_TRACKER.md` і `REMEDIATION_ROADMAP.md`.
