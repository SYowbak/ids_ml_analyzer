# IDS ML Analyzer

IDS ML Analyzer — застосунок для навчання та аналізу IDS-моделей з фокусом на сумісність даних і моделей за природою датасету.

## Основні можливості
- 5 вкладок інтерфейсу: Головна, Тренування, Сканування, Моделі, Історія.
- Явна архітектура Dataset Nature:
  - Network Intrusion: CIC-IDS2017/2018.
  - Classic IDS Benchmark: NSL-KDD.
  - Modern Network Dataset: UNSW-NB15.
- Контроль сумісності файлу і моделі (домен, тип входу, контракт ознак).
- Режими навчання:
  - Автоматичний вибір кандидата (з порівнянням алгоритмів).
  - Ручний/безпечний режими параметрів.
  - Профіль оптимізації для офлайн PCAP-детекції (CIC-IDS).
- Детальний адаптивний звіт сканування (зведення, деталі, графіки ризику, доменні індикатори, хронологія, план дій).
- Локальна історія сканувань у SQLite.

## Що модель показує у звіті (NIDS vs SIEM)
- Базове правило: модель може пояснити лише те, чому була навчена, і що є у вхідних полях.
- Якщо модель навчена як `Normal vs Attack`, у звіті буде факт аномалії без точного класу атаки.
- Якщо модель навчена на деталізованих класах (`DDoS`, `PortScan`, `BruteForce`), у звіті буде назва типу атаки.

### NIDS (CIC-IDS)
- Основний фокус звіту: `src_ip`, `dst_ip`, `src_port`, `dst_port`, `protocol`, часові піки.
- Типові результати: топ джерел атак, порти з найбільшим внеском, мережевий план дій.

### SIEM (NSL-KDD / UNSW-NB15)
- Основний фокус звіту: поведінкові поля домену.
- NSL-KDD: `protocol_type`, `service`, `flag`.
- UNSW-NB15: `proto`, `service`, `state`.
- Типові результати: поведінкові індикатори, пріоритезація triage за сервісами/станами сесій.

### Важливо про формати
- Навчання у поточному UI: тільки `CSV`.
- `PCAP` у поточному UI використовується для сканування/детекції (інференс), а не як прямий формат навчання.

## Вимоги
- Python 3.10+
- Windows/Linux/macOS

## Встановлення
```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

## Запуск
Варіант 1 (рекомендовано на Windows):
```bash
run_app.bat
```

Варіант 2:
```bash
python start_app.py
```

Варіант 3 (напряму Streamlit):
```bash
streamlit run src/ui/app.py
```

## Архітектура
- `src/ui/`: Streamlit UI і активний runtime-потік вкладок.
- `src/core/`: завантаження даних, препроцесинг, рушій моделей, схеми доменів.
- `src/services/`: сервіси БД та налаштувань (поточний UI викликає `DatabaseService` і `SettingsService`).
- `datasets/`: навчальні/тестові CSV, PCAP, user uploads.
- `models/`: збережені моделі.

## Важливо про серіалізацію XGBoost
Для XGBoost використовується збереження бустера у форматі `.ubj` (через `save_model`) замість зберігання повного об'єкта у pickle/joblib. Це знижує ризики несумісності між версіями бібліотек.

## Тестування
```bash
pytest -v
```

## Реальний quality gate навчання
Для перевірки реального навчання/детекції (без pre-demo сценаріїв) використовуйте:

```bash
python scripts/real_training_quality_gate.py
```

Цей скрипт виконує:
- bootstrap обов'язкових CIC моделей з наявних датасетів;
- регресійні тести навчання/калібрування;
- strict real-PCAP E2E перевірку (`IDS_STRICT_E2E=1`).

Примітка: частина real-PCAP E2E тестів залежить від наявності вже навчених моделей у `models/`.
Для strict-профілю (щоб відсутні моделі/файли не skip-ались, а падали):

```powershell
$env:IDS_STRICT_E2E=1
pytest -q -rs tests/test_pcap_real_e2e_regression.py
```
