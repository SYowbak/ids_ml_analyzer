# IDS ML Analyzer - технічна документація

## 1. Статус цієї документації

Цей документ підготовлено після звірки поточного коду проєкту. Базою для опису були:

- `src/ui/app.py` як точка входу чинного потоку виконання;
- активні вкладки `src/ui/tabs/home.py`, `training.py`, `scanning.py`, `models.py`, `history.py`;
- модулі `src/core/*`, які реально викликаються в цих вкладках;
- база даних у `src/database/models.py` та `src/services/database.py`;
- допоміжні скрипти з каталогу `scripts/`.

У документі окремо розділено:

- активну логіку поточної версії;
- модулі, що присутні в репозиторії, але не лежать на основному шляху виконання.

Такий поділ важливий для коректного технічного опису: він дозволяє описати реальну систему без змішування робочого функціоналу з підготовленими або експериментальними компонентами.

## 2. Призначення системи

IDS ML Analyzer - локальний веб-застосунок на Streamlit для навчання та застосування ML-моделей виявлення атак у трьох доменах:

- `CIC-IDS` - мережевий домен `NIDS`;
- `NSL-KDD` - класичний IDS-бенчмарк у режимі `SIEM`;
- `UNSW-NB15` - сучасний мережевий набір даних у режимі `SIEM`.

Система орієнтована на керований і відтворюваний робочий процес:

1. користувач обирає сумісний навчальний набір;
2. навчає модель з контрольованими параметрами;
3. застосовує модель до `CSV` або `PCAP` там, де це дозволено доменом;
4. переглядає ризик, таблицю аномалій, рекомендації та історію запусків.

## 3. Що реально входить до поточної версії

### 3.1 Активні вкладки інтерфейсу

У `src/ui/app.py` активовано саме 5 вкладок:

- `Головна`
- `Тренування`
- `Сканування`
- `Моделі`
- `Історія`

### 3.2 Підтримувані дані та режими

| Сімейство | Режим | Навчання | Сканування | Коментар |
|---|---|---|---|---|
| `CIC-IDS` | `NIDS` | лише `CSV` | `CSV`, `PCAP`, `PCAPNG`, `CAP` | Є окремий профіль оптимізації під офлайн `PCAP` |
| `NSL-KDD` | `SIEM` | лише `CSV` | лише `CSV` | Працює у межах своєї схеми ознак |
| `UNSW-NB15` | `SIEM` | лише `CSV` | лише `CSV` | Працює у межах своєї схеми ознак |

### 3.3 Реально доступні алгоритми

У поточному UI доступні лише:

- `Random Forest`
- `XGBoost`
- `Isolation Forest`

Обмеження:

- `Isolation Forest` дозволений лише для `CIC-IDS`;
- для `NSL-KDD` і `UNSW-NB15` у поточному UI доступні тільки `Random Forest` і `XGBoost`;
- `XGBoost` показується лише якщо доступний у поточному оточенні через `ModelEngine.ALGORITHMS`.

### 3.4 Важлива постановка задачі в поточному UI

У чинній гілці навчання `Random Forest` і `XGBoost` тренуються як бінарні моделі:

- `Normal`
- `Attack`

Активний інтерфейс у цій версії вирішує задачу бінарної класифікації `норма/атака`. Детальні рекомендації щодо типу атаки формуються на етапі інтерпретації результату через каталог загроз (`threat_catalog.py`).

## 4. Архітектура поточного потоку виконання

### 4.1 Активний шлях виконання

Поточний шлях виконання побудований так:

1. `src/ui/app.py` ініціалізує `DatabaseService` і `SettingsService`.
2. Кожна з 5 вкладок викликає власний рендерер.
3. Вкладки `training.py` і `scanning.py` напряму використовують `DataLoader`, `Preprocessor`, `ModelEngine`, схеми доменів і політику порогів.
4. Вкладка `models.py` читає лише підтримувані маніфести через `ModelEngine.list_models(include_unsupported=False)`.
5. Вкладка `history.py` читає історію зі SQLite через `DatabaseService`.

### 4.2 Активні сервіси

У чинному `app.py` реально ініціалізуються два сервіси:

- `DatabaseService` (`src/services/database.py`) — підключення до SQLite, збереження та читання історії сканувань;
- `SettingsService` (`src/services/settings_service.py`) — локальні налаштування користувача.

Додатково під час запуску `app.py` ініціалізується автоматичне очищення завантажених користувачем файлів (`src/services/cleanup.py`).

Інші сервіси, що присутні у репозиторії, не підключені до основного шляху виконання поточної версії (див. підрозділ 5.2).

### 4.3 Логування і конфігурація UI

Під час запуску `app.py` налаштовується логування виконання:

- лог-файл: `reports/logs/runtime.log`;
- рівень логування визначається `IDS_LOG_LEVEL`, за замовчуванням `INFO`.

Файл `.streamlit/config.toml` містить перевірені параметри:

- `maxUploadSize = 2000`
- `enableCORS = false`
- `enableXsrfProtection = false`
- світла тема (`base = "light"`)

## 6. Конвеєр навчання моделі

### 6.1 Кроки, які реально виконує `training.py`

1. Користувач обирає природу датасету.
2. Завантажує один або кілька `CSV`.
3. Система перевіряє, що всі файли належать до сумісного сімейства.
4. Визначається підтверджений `dataset_type`.
5. Користувач обирає режим:
   - `Простий (рекомендовано)`
   - `Експертний (повний контроль)`
6. Для `CIC-IDS` можна ввімкнути опцію `Оптимізувати навчання для виявлення у PCAP`.
7. Викликається `_run_training(...)`.
8. Зберігається модель, маніфест і, для `XGBoost`, окремий файл бустера `.ubj`.

### 6.2 Простий режим

Простий режим:

- не перебирає всі моделі автоматично;
- не є повноцінним автоматичним добором усіх алгоритмів;
- запускає один обраний алгоритм із безпечними рекомендованими параметрами;
- вимикає `GridSearchCV`.

Простий режим дає спрощений контроль над навчанням, але не виконує автоматичне глобальне порівняння всіх алгоритмів.

### 6.3 Експертний режим

Експертний режим відкриває:

- вибір алгоритму;
- `GridSearchCV` для `Random Forest` і `XGBoost`;
- ліміт рядків з одного `CSV`;
- частку тестової вибірки;
- ручні гіперпараметри для конкретного алгоритму.

### 6.4 Логіка для supervised-моделей

У `_train_supervised_model(...)`:

- цільові мітки зводяться до бінарного формату `Normal/Attack`;
- створюється `Preprocessor(dataset_type=dataset_type)`;
- відбувається `train/test split`;
- виконується `ModelEngine.fit(...)`;
- обчислюються `accuracy`, `precision`, `recall`, `f1`;
- підбирається `recommended_threshold`;
- за наявності референсних файлів із `datasets/TEST_DATA` поріг може бути додатково скоригований.

### 6.5 Логіка для `Isolation Forest`

У `_train_isolation_forest(...)`:

- режим дозволений лише для `CIC-IDS`;
- модель навчається на нормальному трафіку;
- потрібна достатня кількість benign-записів;
- використовується `Preprocessor(..., enable_scaling=True)`;
- поріг `if_threshold` калібрується окремо;
- у метаданих моделі фіксується, чи вмикався `PCAP`-орієнтований профіль.

### 6.6 Артефакти навчання

Після навчання зберігаються:

- основний файл моделі `.joblib`;
- супровідний маніфест `.manifest.json`;
- для `XGBoost` - файл бустера `.ubj`.

Типові шаблони імен:

- supervised: `dataset_algorithm_timestamp.joblib`
- `Isolation Forest`: `dataset_isolation_forest_timestamp.joblib`

## 7. Конвеєр сканування

### 7.1 Кроки, які реально виконує `scanning.py`

1. Користувач завантажує `CSV` або `PCAP`.
2. `DataLoader.inspect_file(...)` визначає тип входу, датасет і режим.
3. Для `PCAP` виконується додаткова попередня перевірка на наявність валідних IP-flow ознак.
4. `ModelEngine.list_models(include_unsupported=False)` повертає лише підтримувані моделі.
5. Моделі фільтруються за сумісністю типу входу й домену.
6. Користувач обирає режим вибору моделі:
   - `Автоматично`
   - `Вручну`
7. Система визначає рекомендований поріг або приймає ручне значення чутливості.
8. Викликається `_run_scan(...)`.
9. Результат обчислює ризик, деталі аномалій і записується в історію.

### 7.2 Сумісність і блокування запуску

Сканування може бути заблоковане, якщо:

- файл не відповідає підтримуваному формату;
- немає валідних IP-flow ознак у `PCAP`;
- `CSV` не відповідає контракту ознак моделі;
- модель не пройшла перевірку якості для авто-вибору;
- є жорстка невідповідність природи файлу й моделі.

При невідповідності природи файлу і моделі система попереджає про невідповідність і блокує запуск за замовчуванням, однак користувач може примусово підтвердити запуск кнопкою `Все одно спробувати`.

### 7.3 Обробка даних під час сканування

У `_run_scan(...)` відбувається:

1. завантаження моделі та її метаданих;
2. читання файлу через `DataLoader.load_file(...)`;
3. перетворення ознак через `Preprocessor.transform(...)`;
4. інференс через `ModelEngine`.

Для `Random Forest` і `XGBoost`:

- обчислюються ймовірності атаки;
- порівняння виконується з обраним порогом `sensitivity`;
- результатом є мітка `Normal` або `Attack`.

Для `Isolation Forest`:

- обчислюється `decision_function`;
- формується `effective_threshold`;
- передбачено окремі відновлювальні механізми для випадків сильного зсуву розподілу;
- технічну діагностику можна переглянути в розкривному JSON-блоці.

### 7.4 Розрахунок ризику

У поточному коді:

- `показник ризику = кількість аномалій / загальна кількість записів * 100`

Рівні ризику:

- `НИЗЬКИЙ`, якщо менше `15`
- `СЕРЕДНІЙ`, якщо від `15`
- `ВИСОКИЙ`, якщо від `40`
- `КРИТИЧНИЙ`, якщо від `75`

### 7.5 Що повертає сканування

Результат містить:

- назву файлу;
- сімейство даних і режим аналізу;
- алгоритм і назву моделі;
- кількість записів;
- кількість аномалій;
- показник ризику та рівень ризику;
- таблицю результатів;
- розподіл передбачень;
- за наявності `Isolation Forest` - технічну діагностику порогів.

### 7.6 Експорт

У поточному UI доступні лише:

- `JSON`-звіт з підсумковими полями;
- `CSV` з аномальними записами.

## 8. База даних і збереження стану

### 8.1 Файл бази даних

За замовчуванням база даних створюється в корені проєкту як:

- `database/ids_history.db`

За потреби шлях може бути перевизначений через ключ `db_path` у `src/services/user_settings.json`.

### 8.2 Таблиці

У `src/database/models.py` визначено 3 ORM-моделі, що відповідають трьом таблицям:

- `analysis_sessions` — записи сесій аналізу (файл, статус, кількість записів, аномалії, ризик, час обробки);
- `detected_anomalies` — виявлені аномалії з деталізацією (IP-адреси, порти, тип атаки, впевненість);
- `trained_models` — навчені моделі (назва, тип, метрики якості, шлях до файлу).

### 8.3 Що зберігається у налаштуваннях

Файл `src/services/user_settings.json` у поточному репозиторії вже містить:

- `anomaly_threshold`
- `ui_language`
- `ui_theme`

Під час сканування активне значення порогу може бути повторно записане у `anomaly_threshold`.

## 9. Сценарії використання

### 9.1 Сценарій 1 - навчання NIDS-моделі для `CIC-IDS`

1. Відкрити вкладку `Тренування`.
2. Обрати природу `Мережева інфільтрація`.
3. Завантажити один або кілька сумісних `CIC-IDS CSV`.
4. Обрати `Random Forest`, `XGBoost` або `Isolation Forest`.
5. За потреби ввімкнути `Оптимізувати навчання для виявлення у PCAP`.
6. Запустити навчання.
7. Перевірити появу артефактів у `models/`.
8. За потреби перейти до вкладки `Моделі` і зробити модель типовою.

### 9.2 Сценарій 2 - навчання SIEM-моделі для `NSL-KDD` або `UNSW-NB15`

1. Відкрити вкладку `Тренування`.
2. Обрати відповідну природу.
3. Завантажити лише сумісні `CSV`.
4. Обрати `Random Forest` або `XGBoost`.
5. Навчити модель.
6. Використати її для сканування `CSV` цього самого домену.

### 9.3 Сценарій 3 - сканування підозрілого `PCAP`

1. Відкрити вкладку `Сканування`.
2. Завантажити `PCAP`.
3. Дочекатися попередньої перевірки на IP-flow.
4. Переконатися, що доступна сумісна модель `CIC-IDS`.
5. Залишити автоматичний поріг або налаштувати його вручну.
6. Запустити сканування.
7. Переглянути ризик, таблицю аномалій і за потреби експортувати `JSON` або `CSV`.

### 9.4 Сценарій 4 - супровід моделей і журналу запусків

1. Вкладка `Моделі` дає змогу:
   - фільтрувати моделі;
   - порівнювати метрики;
   - встановити типову модель;
   - видаляти моделі та пов'язані артефакти;
   - завантажувати `.joblib`.
2. Вкладка `Історія` дає змогу:
   - переглядати останні до 200 записів;
   - бачити кількість алертів і середній ризик;
   - очищати історію після підтвердження.

## 10. Верифікація та контроль якості

### 10.1 Що є в проєкті

У корені репозиторію немає активної папки `tests/`. Натомість є робочі перевірки в `scripts/`:

- `runtime_smoke_quality_checks.py`
- `real_training_quality_gate.py`
- `bootstrap_required_models.py`
- `benchmark_multifamily_algorithms.py`

### 10.2 Що перевіряють скрипти

`runtime_smoke_quality_checks.py` перевіряє:

- цикл навчання;
- калібрування `Isolation Forest`;
- політику порогів;
- реальні перевірки на `PCAP`, якщо доступні потрібні дані.

`real_training_quality_gate.py` виконує:

- bootstrap потрібних моделей;
- `compileall` для `src/` і `scripts/`;
- швидкі перевірки виконання у строгому режимі.

`benchmark_multifamily_algorithms.py` навчає моделі для доступних сімейств і проганяє їх на контрольних файлах з `datasets/TEST_DATA`.

## 11. Обмеження поточної версії

### 11.1 Функціональні обмеження

- навчання у UI працює лише з `CSV`;
- `PCAP` використовується лише для сканування;
- `Isolation Forest` підтримується лише для `CIC-IDS`;
- активний шлях supervised-навчання є бінарним, а не мультикласовим;
- у списку моделей показуються лише ті артефакти, які пройшли перевірки `manifest_version`, домену та алгоритму.

## 12. Висновок

IDS ML Analyzer — робоча локальна система виявлення кіберзагроз із чітко визначеним активним потоком: навчання на `CSV`, контрольоване сканування `CSV/PCAP`, керування моделями та збереження історії запусків. Модулі, не підключені до поточного інтерфейсу (підрозділ 5.2), залишені як архітектурний резерв для подальшого розвитку.
